# Anti-DDoS защита для Minecraft сервера

***

### 1. Что такое DDoS атака

**DDoS (Distributed Denial of Service)** — атака при которой злоумышленник направляет огромное количество ложных запросов с множества устройств одновременно. Цель — исчерпать ресурсы сервера или канал связи так чтобы реальные игроки не могли подключиться.

#### Почему Minecraft серверы атакуют

* Конкурентная борьба между владельцами серверов
* Вымогательство — требование денег за остановку атаки
* Месть забаненных игроков
* Доступность инструментов — DDoS-стрессеры доступны от $5/час
* Открытость порта 25565 — легко обнаружить сканированием

***

### 2. Уровни сетевой модели OSI

Понимание уровней OSI критично для выбора защиты:

| Уровень | Название     | Протоколы          | Что атакуют                       |
| ------- | ------------ | ------------------ | --------------------------------- |
| L3      | Сетевой      | IP, ICMP           | IP флуд, спуфинг, амплификация    |
| L4      | Транспортный | TCP, UDP           | SYN флуд, UDP флуд                |
| L7      | Прикладной   | Minecraft protocol | Bot-join, login flood, ping flood |

> **Важно:** UDP и TCP оба находятся на L4. Разница в том что TCP требует handshake (SYN→SYN-ACK→ACK) что позволяет фильтровать некорректные соединения. UDP не имеет handshake — пакеты летят без установки соединения, что усложняет фильтрацию.

***

### 3. Виды атак

#### L3/L4 атаки (сетевой и транспортный уровень)

**UDP Flood** Массовая отправка UDP пакетов на порт сервера. Самый распространённый тип. Мощность: от 1 Gbps до 2+ Tbps. Фильтруется на уровне хостинга.

**SYN Flood** Тысячи незавершённых TCP соединений. Сервер держит открытые сессии до таймаута, исчерпывая память. Фильтруется SYN cookies на L4.

**ICMP Flood / Amplification** Использование DNS, NTP серверов для усиления атаки. Коэффициент усиления: DNS до 100x, NTP до 556x. Фильтруется на L3.

#### L7 атаки (прикладной уровень)

**Bot-join атака** Ботнет подключается к серверу, проходит TCP handshake и Minecraft handshake, симулируя реальных игроков. Проходит через L3/L4 фильтры как легитимный трафик.

**Login Flood** Массовые попытки входа с разными именами. Нагружает авторизацию и базу данных сервера.

**Ping Flood (Server List Ping)** Тысячи запросов MOTD без реального намерения подключиться. Нагружает логику сервера.

**Fake Player Flood** Боты входят на сервер и имитируют поведение игроков. Самый сложный для фильтрации тип.

### 4. Как работает защита

#### Scrubbing-центры

Специализированные узлы которые пропускают через себя весь трафик, отделяют вредоносные пакеты от легитимных и передают на сервер только чистый трафик.

#### BGP Anycast

Один IP адрес анонсируется из нескольких географических точек одновременно. Трафик автоматически идёт к ближайшему узлу, нагрузка при атаке распределяется между десятками точек по всему миру.

#### Rate Limiting

Ограничение количества пакетов или соединений с одного IP за единицу времени. Легитимный игрок генерирует несравнимо меньше трафика чем атака.

#### Always-On vs On-Demand

| Тип       | Mitigation time | Для Minecraft |
| --------- | --------------- | ------------- |
| Always-On | < 1 сек         | ✅ Обязателен  |
| On-Demand | 30–300 сек      | ❌ Неприемлемо |

> Для игровых серверов нужен исключительно Always-On режим. За 30–300 секунд пока включается On-Demand защита сервер уже недоступен для игроков.

### 5. Схема защиты

#### Архитектура с единым IP хостинга

На большинстве хостингов все серверы находятся на **одном общем IP** но разных портах. В такой архитектуре IP изначально публичный и скрывать его нет смысла. Защита строится не на сокрытии адреса а на фильтрации трафика.

{% hint style="warning" %}
Схема представлена на примере с проектами Shard Team начиная с 2026 года.
{% endhint %}

```
┌─────────────────────────────────────────────────────────────────┐
│                  СХЕМА ЗАЩИТЫ MINECRAFT СЕРВЕРА                 │
└─────────────────────────────────────────────────────────────────┘

  [🎮 Игрок]                        [💀 DDoS Ботнет]
  TCP :25565                         UDP Flood / SYN Flood
  UDP :24454 (SVC)                   ~20 Gbps · 800 Mpps
       │                                    │
       └──────────────────┬─────────────────┘
                          │ Весь трафик на один IP
                          ▼
          ┌───────────────────────────────┐
          │            ХОСТИНГ            │
          │        Anti-DDoS L3/L4        │
          │                               │
          │  ✅ UDP Flood    → BLOCK      │
          │  ✅ SYN Flood    → BLOCK      │
          │  ✅ ICMP Flood   → BLOCK      │
          │  ⚠️  L7 атаки    → PASS       │
          └───────────────────────────────┘
                          │
                          │ Чистый трафик (L7 может проходить)
                          ▼
          ┌───────────────────────────────┐
          │        GATE LITE PROXY        │  ← Опционально
          │        L7 Фильтрация          │
          │                               │
          │  ✅ Bot-join flood  → BLOCK   │
          │  ✅ Login flood     → BLOCK   │
          │  ✅ Ping flood      → BLOCK   │
          │  ✅ Fake players    → BLOCK   │
          │  ✅ Реальный игрок  → PASS    │
          └───────────────────────────────┘
                          │
                          │ Только верифицированный TCP трафик
                          ▼
          ┌───────────────────────────────┐
          │            SERVER             │
          │     Minecraft + Плагины       │
          │                               │
          │  TCP :25565  ← через Gate     │
          │  UDP :24454  ← напрямую ⚠️    │  ← SVC уязвимость
          └───────────────────────────────┘
                          ▲
                          │ SVC UDP :24454 напрямую
                    [🎮 Игрок]
```

#### Что защищает каждый уровень

| Уровень            | Что блокирует                                   | Что пропускает           |
| ------------------ | ----------------------------------------------- | ------------------------ |
| Хостинг L3/L4      | UDP flood, SYN flood, ICMP, амплификация        | L7 атаки, корректный TCP |
| Gate Lite L7       | Bot-join, login flood, ping flood, fake players | Верифицированных игроков |
| Плагины на сервере | Оставшиеся боты, абьюз механик                  | Легитимные действия      |

### 8. Simple Voice Chat и UDP защита

#### Проблема

Simple Voice Chat (SVC) работает на **UDP порту 24454**. Gate Lite и большинство прокси работают только с TCP и не могут проксировать UDP трафик голосового чата. SVC соединяется напрямую между клиентом и сервером минуя Gate.

#### Что защищает хостинг

Хостинг с L3/L4 защитой **блокирует:**

* UDP flood на порт 24454
* Простые флуд атаки на транспортном уровне

Хостинг **не блокирует:**

* L7 атаки имитирующие легитимные SVC пакеты (требует реверс-инжиниринга протокола, редкий сценарий)

#### Вывод по SVC

Для большинства реальных угроз L3/L4 защиты хостинга **достаточно**. Целенаправленная L7 атака именно на протокол SVC — крайне специфичный сценарий который на практике почти не встречается.

Полноценно защитить UDP SVC без собственной инфраструктуры (отдельный сервер, VPN туннель) **невозможно**.

### 9. Gate Lite Proxy — нужен ли он

#### Когда Gate Lite нужен

Gate Lite имеет смысл если хостинг даёт только L3/L4 защиту и существует реальная угроза L7 атак:

* Bot-join флуд
* Login flood
* Ping flood (Server List Ping)
* Fake players

Gate Lite понимает протокол Minecraft и отсеивает ботов **до того** как они нагружают основной сервер. Это особенно важно для тяжёлых гибридных ядер типа Mohist/Magma которые быстрее падают под нагрузкой.

#### Когда Gate Lite не нужен

* Атак на L7 не было и реальная угроза низкая
* Хостинг уже имеет встроенную игровую L7 фильтрацию
* Важна минимальная задержка (Gate добавляет 1–5 мс)

#### Итоговая логика

```
Хостинг L3/L4  →  закрывает флуд атаки
Gate Lite L7   →  закрывает умные атаки на протокол Minecraft
Плагины        →  последний рубеж на самом сервере
```

Каждый уровень закрывает то что не умеет предыдущий. Добавлять прокси **между** P2G и Gate Lite смысла нет — это лишняя задержка без пользы.

### 10. Выбор защиты по размеру сервера

#### Маленький сервер (до 50 игроков)

* Хостинг с встроенной L3/L4 защитой (Play2GO, OVH Game, BisectHosting)
* TCPShield Free Tier как дополнительный L7 слой
* Антибот плагин на сервере
* **Бюджет:** $0–20/мес

#### Средний сервер (50–500 игроков)

* OVH Game Server — 160 Gbps Always-On защита включена в цену
* Gate Lite Proxy для L7 фильтрации
* Антибот плагин совместимый с ядром
* **Бюджет:** $20–80/мес

#### Крупный сервер (500+ игроков)

* Path.net или Voxility — реальная защита 1+ Tbps с SLA гарантиями
* BGP Anycast для минимальной задержки по всему миру
* Выделенный специалист по сетевой безопасности
* Регулярный аудит и кастомные правила фильтрации
* **Бюджет:** $100–500+/мес

#### Универсальные правила

* Использовать **Always-On** защиту, не On-Demand
* Выбирать провайдера с узлами в регионе игроков
* Тестировать задержку до покупки
* Несколько уровней защиты надёжнее одного мощного
